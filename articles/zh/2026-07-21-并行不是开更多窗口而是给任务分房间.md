# 并行不是开更多窗口而是给任务分房间

> 发布日期:2026-07-21 · [中文](../zh/2026-07-21-并行不是开更多窗口而是给任务分房间.md) | [English](../en/2026-07-21-parallel-work-needs-separate-rooms.md)

---

很多人正在发现一种新的野心。

不是“我想把这件事做快一点”。

那只是 AI 的第一层。

新的野心是：“我想让十件有用的事，同时往前走。”

一个 Agent 清理旧模块，一个 Agent 修测试，一个 Agent 改页面，一个 Agent 查一个小市场，一个 Agent 把笔记整理成草稿，一个 Agent 做一个也许本周就会被砍掉的小工具。

乍一看，这很像丰盛。过去的瓶颈是人的手不够。现在手变便宜了。再开一个会话，再开一个窗口，再给机器派一件活儿。让硅基时间先跑起来，碳基时间该睡就睡。

可是，并行一旦变便宜，真正的瓶颈就从“做不做得完”变成了“装不装得住”。

这些任务能不能保持分离？系统能不能记住谁在改什么？你能不能看出哪个结果值得留下？一条线失败了，能不能不污染另一条线？你回来验收的时候，能不能不用把一团乱麻重新塞回自己脑子里？

如果答案是否定的，那就不叫并行工作。

那叫并行混乱。

所以，Git 里一个老工具，今天突然有了很强的哲学意味：Worktree。

Git 官方文档对 worktree 的解释很直接：它可以管理附属于同一仓库的多个工作区。通俗说，同一个代码仓库，可以同时拥有多个工作目录，让不同分支或实验在不同地方展开。[^1] GitHub 在 2015 年介绍 Git 2.5 时，也把新的 `git worktree` 命令描述成：在同一个仓库上创建额外的工作树，避免维护多份克隆仓库带来的同步成本。[^2]

这个事实是技术的。

但它背后的道理更大。

分支是路线，Worktree 是房间。

分支告诉你：这条工作线要往哪个方向走。房间告诉你：这件事到底在哪里发生。

在 AI 之前，很多人只靠“路线”也能活下来。你做一件事，切分支，换上下文，改完再切回来。你用自己的脑子记住现场，用自己的耐心支付切换成本。麻烦是麻烦，但人的速度本身限制了混乱的规模。

AI 改变的是混乱的放大速度。

当一个人可以同时启动五个、十个、二十个任务，如果没有独立房间，代价会很快复利。一件本该隔离的小修补，摸到了共享文件。一个临时修改，破坏了大重构的假设。两个 Agent 各自做了局部合理的决定，合起来却全局不合理。你过几个小时回来，甚至分不清哪一处改动属于哪一个意图。

问题不在于 AI 不适合并行。

问题在于，我们常常还没设计“并行空间”，就急着要“并行产出”。

这件事不只发生在编程里。

写作里的并行空间，是不同论点有不同草稿，而不是所有半成形的想法挤在同一个文档里。学习里的并行空间，是每条探索线都有自己的项目和记录。做生意里的并行空间，是每个实验有自己的用户、承诺和验收标准，而不是十个想法塞进一个含糊计划。

房间一旦缺失，聪明就会漏出去。

你可以有很强的模型，很长的上下文，很漂亮的 prompt，最后结果仍然很差。原因很简单：工作没有边界。

这句话很多人不爱听。因为大家希望 AI 把自己从结构里解救出来。可现实恰恰相反：AI 奖励结构。它会让有结构的人更强，也会让没结构的人更吵。

这也是“用 AI 让人变懒”和“用 AI 让人变强”的分界线。

变懒的用法是：我不想想清楚任务，所以交给模型全权处理。

变强的用法是：我先更认真地想清楚任务的形状，再让模型在一个更小、更干净、更容易验收的房间里工作。

前者制造动静。

后者制造杠杆。

而杠杆永远需要容器。

水进入水渠，才会推动水轮。电进入电路，才真的有用。没有容器，更多能量常常只意味着更大破坏。

AI 是能量。

Worktree 只是其中一种小容器。

更深一层的容器，其实有四个部分。

第一，任务需要边界。

这件事允许碰哪里？明确不许碰哪里？如果任务叫“优化一下产品”，房间太大。如果任务叫“只替换某个页面的空状态文案，不改布局”，房间就小到可以检查。

第二，任务需要终点。

不是感觉，不是“做得更好一点”，而是终点。测试通过，页面能打开，草稿达到某种形状，报告包含三条有来源的发现，原型可以被另一个人打开。没有终点，并行工作会变成无限工作，因为每条线都不知道什么时候该停。

第三，任务需要合并规则。

并行不等于接受。Agent 完成了一件事，不代表它应该被合并。一篇草稿有意思，不代表它属于今天的文章。一个功能能跑，不代表它让产品变好。如果所有产出都被当成进展，系统最后只会堆满杂物。

第四，任务需要恢复路径。

这条线失败了怎么办？能不能干净丢掉？能不能和另一条路线对比？能不能留下有用部分，扔掉剩下部分？好的房间有门。坏的房间会变成陷阱。

这些听起来都很普通。

正因为普通，才重要。

AI 时代总是把我们推向一些很大的词：Agent、Graph、编排、自治团队、自进化系统。我喜欢这些词，前提是它们指向真实机制。可真实机制常常从很朴素的地方开始：给每一块工作一个地方，一个边界，一个终点，一个合并规则，以及一条退出路径。

也因此，过去那种“技术人”和“非技术人”的区分正在变得没那么有用。即使你永远不敲 `git worktree`，这个概念也值得理解。问题已经不只是“你会不会编程”，还包括：“你能不能安排工作，让智能在里面运行，同时不毁掉自己的上下文？”

这是工程问题，但不只属于工程师。

老师设计练习，需要房间。创作者测试内容方向，需要房间。小老板尝试 offer，需要房间。学习者探索 AI 工具，也需要房间。

表面工具不同，底层纪律一样：把工作分离到足够清楚，让反馈可以被读懂。

这才是并行真正的意义。不是开更多窗口，而是在单位时间里制造更多干净反馈。

如果五个 Agent 跑了五个小时，你却需要十个小时才能理解它们到底做了什么，那你其实没有赚到多少。如果三条小线独立运行，每条线回来都给你一个清晰的“行、不行、也许值得继续”，你的学习速度才真的提高了。

目标不是最大动静。

目标是最大干净信号。

“干净”这两个字很关键。

混乱的信号比没有信号更危险，因为它看起来像证据，里面却藏着糊涂。三个实验同时改了同一个条件，你就没法知道到底是谁产生了效果。一个重构里混进了无关功能，你就没法判断重构本身好不好。一篇文章同时塞进两个论点，你就没法判断哪个论点真正成立。你没定义什么叫帮上忙，就永远不知道 AI 到底有没有帮上忙。

所以，在让 AI 做更多之前，先问一个很无聊的问题：

这件工作住在哪里？

这个问题逼你给任务命名，逼你把意图和情绪分开，逼你决定什么叫完成，逼你保护系统里的其他部分，也逼你像一个管理不止一双手的人那样思考。

这就是为什么“一人团队”这个说法既真实，又危险。

真实，是因为 AI 确实给了一个人很多临时工。危险，是因为很多人听到“团队”只想到劳动力，却忘了管理。没有角色、空间、标准和审查的团队，不是团队，只是一群人。

AI 可以很快给你一群人。但你得自己把它建成团队。而建团队这件事，在第一条 prompt 之前就已经开始了。

这件事也回到我一直相信的一个长期主义判断：真正的成长，不只是增加能力，而是学会容纳更多能力，同时不被它扭曲。

更多记忆，需要更好的召回。更多工具，需要更好的权限。更多产出，需要更好的品味。更多自动化，需要更好的恢复。更多 Agent，需要更好的房间。

每一种力量的增加，都在要求对应的结构升级。

忽视这一点，AI 只会变成制造焦虑的新机器。

尊重这一点，另一种可能才会出现。一个人不只是工作更快，而是能同时运行几条干净的学习循环。不是假装自己是机器，也不是把判断力交给机器，而是设计环境，让机器的工作真的变成人的进步。

所以，更好的问题不是“我能同时开多少个 Agent？”

而是“我能同时维护多少条独立、可理解、可验收、可回滚的工作线？”

这才是更好的强大指标。

因为未来不会奖励制造最多动静的人。

未来会奖励那些能制造最多干净动静的人：留下信号，扔掉噪音，明天回来时，系统比今天更强一点。

并行工作，不是从更多窗口开始的。

它从给任务分房间开始。
